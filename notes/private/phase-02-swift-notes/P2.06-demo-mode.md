# P2.06 â€” Demo mode (TS-developer field notes)

What a TS reviewer needs to know to read `DemoConfig.swift`, `DemoCycleDriver.swift`, and the demo wiring in `MenubarApp.swift` honestly. Concepts that show up in this PR only.

## `ProcessInfo.processInfo.environment` and `.arguments`

```swift
let envOn = environment["CODOGOTCHI_DEMO"] == "1"
let argOn = arguments.contains("--demo")
```

- `ProcessInfo.processInfo.environment` is `[String: String]` â€” Swift's view of the process's POSIX environment block at launch. It is a snapshot, not a live binding; `setenv` after launch will not appear here unless you re-read.
- `ProcessInfo.processInfo.arguments` is `[String]` â€” `argv` for the process. Element `0` is the binary path; user-supplied launch args start at `[1]`.
- TS analogue: `process.env` / `process.argv`. Same shape, same gotchas (env reads at startup, args include the binary path).
- The pure helper `DemoConfig.from(environment:arguments:)` takes both as plain parameters so unit tests can drive it deterministically without setting real env vars. `DemoConfig.forLaunch()` is the thin production seam that calls `ProcessInfo` and forwards.

## Atomic file writes â€” `.atomic` write option

```swift
try data.write(to: sandboxedPath, options: .atomic)
```

- `Data.write(to:options:.atomic)` writes the bytes to a sibling temp file in the same directory and renames it into place on success. Readers that open the destination path either see the old bytes or the new bytes â€” never a half-written file.
- This mirrors the Phase 01 hook's pattern (`write to tmp, rename`). Demo mode therefore exercises the same race-free read semantics that live polling (P2.07) will depend on, instead of inventing a parallel path.
- The rename is filesystem-atomic only inside the same filesystem volume. `$TMPDIR/codogotchi-demo/state.json` and its sibling temp both live under `$TMPDIR`, so the rename stays on one volume by construction.
- TS analogue: `await fs.writeFile(`${path}.tmp`, data); await fs.rename(`${path}.tmp`, path)`. The Swift API folds both into one call.

## Bundle resources vs. fixture paths

```yaml
resources:
  - path: Resources
    optional: true
  - path: Fixtures/state-json
    type: folder
```

- xcodegen's `resources` array enumerates folders/files copied into the built `.app` under `Contents/Resources/` at build time.
- `type: folder` preserves the directory shape â€” `Fixtures/state-json/idle.json` becomes `Resources/state-json/idle.json` in the bundle. Without `type: folder`, xcodegen flattens the JSON files directly into `Resources/`, losing the namespace.
- At runtime: `Bundle.main.resourceURL?.appendingPathComponent("state-json", isDirectory: true)`.
- Tests resolve fixtures via `#file` traversal instead of `Bundle.main` because the unit-test bundle does not embed the demo fixtures â€” there is no point shipping them twice. Production and tests use the same fixture bytes, just different resolvers.

## Dependency injection for the polling-target seam

`MenubarApp` does not hand a hardcoded URL to the renderer or to a polling driver. It builds a `DemoConfig` first, then dispatches:

- demo mode: pass `config.pollingTarget` to `DemoCycleDriver` as `sandboxedPath` and start the cycle.
- live mode (future P2.07): pass `config.pollingTarget` to whatever live polling driver lands then.

That single seam is the entire reason the work split between P2.06 and P2.07 is honest. Both modes converge on the same `pollingTarget` URL; only the *origin* of the bytes at that URL differs (fixture copy vs. real hook). The renderer never sees the difference.

## Test seam: `tickForTesting()` over real `Timer`

`DemoCycleDriver` accepts a `tickInterval` (default `3.0` seconds) and exposes a synchronous `tickForTesting() throws` entrypoint. Tests drive the cycle without scheduling a real timer:

```swift
for _ in 0..<4 { try driver.tickForTesting() }
XCTAssertEqual(observed, [.idle, .implementing, .runningTests, .celebrating])
```

The ticket review focus called this out explicitly â€” real timers with short intervals would work for a two-point ticket, but every test would have to wait wall-clock time and `xcodebuild ... test` would become flaky. Synchronous tick is cleaner.

## Strong-reference holds on `MenubarApp`

```swift
var demoDriver: DemoCycleDriver?
var demoConfig: DemoConfig?
```

Same pattern as `statusItem` and `renderer` (see P2.05 notes): the app delegate keeps a strong reference so the `Timer`'s closure block (which holds `[weak self]`) cannot deallocate the driver mid-cycle. `applicationWillTerminate(_:)` calls `demoDriver?.stop()` to cancel the timer cleanly on quit; without that, the timer block can fire one extra time during shutdown against a partially-torn-down process.
