# P2.08 Swift Notes — Transition Log

Companion to `apps/menubar/Sources/TransitionLog.swift`. Captures the
ground-truth Swift behavior that drove the implementation choices so a future
contributor does not re-derive them under deadline pressure.

## Appending with `FileHandle`

NDJSON is intentionally append-only — each line is one event, the file is
read top-to-bottom forever, and lines must never be rewritten in place. The
two patterns Swift exposes for append-mode writes both work, but they have
different failure modes:

- `FileHandle(forWritingTo: url)` opens an existing file for writing. It
  does **not** open at the EOF — callers must `seekToEnd()` before each
  write, or the new bytes overwrite the start of the file. `TransitionLog`
  takes this path and reopens the handle per write. The cost is one open
  syscall per line; the gain is straightforward crash semantics and trivial
  rotation (we own the file at the moment we close it).
- `FileHandle` with `O_APPEND` opened via the lower-level POSIX API would
  let the kernel atomically append even with concurrent writers. We do not
  need that — there is one writer (`MenubarApp`'s main actor) — so the
  higher-level API is sufficient.

`FileHandle` errors are thrown via Swift's `throws` bridge starting in
macOS 10.15+; we catch and `NSLog` rather than crash. The transition log is
best-effort observability, not the source of truth.

## ISO-8601 timestamps

`ISO8601DateFormatter` is the canonical formatter but is **slow to
initialize** (sub-millisecond, but enough to matter at high frequencies).
The well-known fix is to keep a single static instance — `TransitionLog`
exposes it as `private static let isoFormatter` so every line write reuses
the same configured object. `formatOptions = [.withInternetDateTime,
.withFractionalSeconds]` is what produces `2026-05-20T14:32:11.123Z`; without
`.withFractionalSeconds` the trailing milliseconds get dropped.

Avoid `DateFormatter` with an explicit `dateFormat` template that includes
fractional-second tokens — it works but is locale-sensitive (`en_US_POSIX`
is the safe locale), the template tokens vary by macOS version, and it is
slower than `ISO8601DateFormatter`. The product plan already commits to
ISO-8601, so the canonical formatter is the right hammer.

## Injectable clock

The behavior contract says "no transitions for >60 minutes triggers a
heartbeat." Testing that honestly without `Thread.sleep(60 * 60)` requires
control over what the implementation considers "now."

The chosen seam is a `clock: @escaping () -> Date` closure on the
initializer. The default is `Date.init` (which is `Date()` under the hood),
so production code reads as if there were no seam at all. Tests pass a
captured `var now: Date` and mutate it between calls, then invoke
`tickHeartbeatForTesting()` to run one heartbeat probe synchronously
without scheduling a real `Timer`.

This is the same shape used by `LivePollingDriver.tickForTesting()` and
`DemoCycleDriver.tickForTesting()` — keep the family resemblance so test
authors do not have to learn a new idiom per driver.

## Heartbeat timer probe interval

The product spec says "every 60 minutes." The implementation uses a 60-second
probe `Timer` that consults the injected clock and only emits when the
real elapsed time has crossed the heartbeat interval. The reason for the
finer probe cadence: a single 60-minute `Timer` would drift badly across
sleep/wake, and the wall-clock elapsed time on resume might already exceed
the interval. Probing once a minute keeps the timer trivial while still
hitting the contract within ~1 minute of the configured window.

## Rotation accounting

The ticket flagged heartbeat-vs-rotation accounting as an explicit decision
point. The chosen behavior: **heartbeats count toward rotation accounting**
because every appended byte sits in the same file on disk — excluding
heartbeats from the size calculation would be arithmetic on a number the
filesystem disagrees with. The unit test reflects this by writing
transitions to trigger rotation; heartbeat-only tests do not separately
assert rotation behavior.

Rotation itself is intentionally simple: when a write would extend a file
already past 10 MB, rename the existing log to `state-transitions.log.1`
(overwriting any prior `.1`) and open a fresh `state-transitions.log`.
Single backup; no `.log.2` or `.log.3`. A crash mid-rotation leaves either
the old `.log` (rename failed) or the renamed `.log.1` plus a missing
`.log` (rename succeeded; the next write recreates `.log`) — never an empty
`.log` with lost data, because rename is atomic on local macOS volumes.
