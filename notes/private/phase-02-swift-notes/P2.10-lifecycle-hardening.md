# P2.10 Swift Notes — Lifecycle Hardening

Companion to `apps/menubar/Sources/MenubarApp.swift` for the wake-from-sleep
observer and the `applicationWillTerminate(_:)` cleanup. Captures the AppKit
notification semantics a TS-leaning contributor would otherwise have to
re-derive from `developer.apple.com`.

## Two notification centers, not one

AppKit ships **two** `NotificationCenter` instances and they are not
interchangeable:

- `NotificationCenter.default` — the process-wide bus for in-app
  notifications (`NSApplication`, `NSWindow`, `NSView`, etc.). This is what
  `applicationDidFinishLaunching` and `applicationWillTerminate` ride on,
  delivered to the `NSApplicationDelegate` via the AppKit run loop.
- `NSWorkspace.shared.notificationCenter` — a **separate** center owned by
  the workspace process, used for system-level events the application
  doesn't directly own: sleep/wake (`willSleepNotification`,
  `didWakeNotification`), screen lock/unlock, user session switch, mounted
  volumes, etc.

Wake notifications are workspace events, so the observer must be registered
against the workspace center. Registering on `NotificationCenter.default`
for `NSWorkspace.didWakeNotification` is a silent no-op — the name is
visible but nothing ever posts to that bus.

In TS terms: it's roughly the difference between a process-local
`EventEmitter` and an OS-level event source like a system signal handler.

## Block-based observers vs. selector observers

`NotificationCenter` has two styles of `addObserver`:

```swift
// Selector style — observer is the target; AppKit messages it weakly.
NotificationCenter.default.addObserver(
    self, selector: #selector(handleWake(_:)),
    name: NSWorkspace.didWakeNotification, object: nil)

// Block style — returns an opaque token that owns the block.
let token = NSWorkspace.shared.notificationCenter.addObserver(
    forName: NSWorkspace.didWakeNotification,
    object: nil, queue: .main) { [weak self] _ in
    self?.livePollingDriver?.pollNow()
}
```

`MenubarApp` uses the block style because there is exactly one wake hook
and it does exactly one thing — wiring up an `@objc` selector and a target
function for that would be more ceremony than the call site is worth.

The block-style API returns an **opaque token** (`NSObjectProtocol`). That
token is the only handle you can use later to detach the observer; the
block itself is retained by the notification center until the token is
released or `removeObserver` is called with it. Drop the token without
calling `removeObserver` and the block leaks for the lifetime of the
notification center (the workspace center, in this case, lives as long as
the app process).

## Why the token is a stored property

`var workspaceWakeObserver: NSObjectProtocol?` is held strongly on the
`MenubarApp` instance for two reasons:

1. **Lifetime.** Same reason `menuBuilder`, `renderer`, `demoDriver`,
   `livePollingDriver`, and `transitionLog` are stored properties. The
   `applicationDidFinishLaunching` callback's stack frame goes away the
   moment it returns; anything held only on that stack is eligible for
   ARC to deallocate.
2. **Removal.** `removeObserver(_:)` needs the original token to detach
   the observer cleanly during `applicationWillTerminate`. Without the
   stored reference, there is no handle.

The closure captures `self` weakly (`[weak self]`) so the observer block
does not keep `MenubarApp` alive past its natural lifetime, and the
observer guard inside `applicationWillTerminate` flips the token to `nil`
after detaching to make the cleanup idempotent.

## Why no sleep observer

There is no `NSWorkspace.willSleepNotification` handler. The reason is
narrow and worth writing down explicitly so future contributors don't
"helpfully" add one:

- `Timer` does not fire while the system is asleep. `tickInterval = 1.0`
  pauses for the duration of sleep and resumes when the run loop resumes
  — no explicit pause is needed.
- File I/O against `~/.codogotchi/state.json` is a one-shot read per
  tick. There are no long-lived file descriptors that need draining
  before sleep. (The transition-log file handle is append-only and
  survives sleep cleanly — see P2.08 notes.)
- Subprocesses: none.

The wake handler exists because polling lag is observable to the user
(the pet stays on the pre-sleep state for up to a second after wake until
the next tick). The pre-sleep symmetry is not observable; adding a sleep
handler would be code without behavior.

## `applicationWillTerminate(_:)` cleanup semantics

`applicationWillTerminate(_:)` is AppKit's standard "the user (or the OS)
asked us to quit, finish up." It runs on the main run loop, synchronously,
before the process exits. Anything that needs to flush — file handles,
sockets, subprocesses, observer tokens, system resources — should be
released here.

The contract is best-effort, not guaranteed:

- It runs on graceful quit (⌘Q, **Quit Codogotchi**, `NSApplication.terminate`,
  logout, shutdown).
- It does **not** run on a crash, on `kill -9`, or on a forced quit
  (Activity Monitor → Force Quit, or `SIGKILL`).

In practice that means cleanup should be defensive — anything truly
critical needs to be safe under "process died abruptly" too. The current
cleanups in `MenubarApp.applicationWillTerminate` are:

1. Remove the workspace wake observer (idempotent).
2. Stop the `DemoCycleDriver` timer (no-op if not in demo mode).
3. Stop the `LivePollingDriver` timer (no-op if `Timer` was never
   created).
4. Stop the `TransitionLog` writer — invalidates the heartbeat timer.
   The log does **not** keep a long-lived file handle: every write opens
   a `FileHandle`, appends the NDJSON line, and closes it under a
   `defer` (see `TransitionLog.swift`), so there is no descriptor to
   release at shutdown.

All four are safe to call regardless of whether the underlying resource
was ever activated, and all four are also safe under abrupt death because
each underlying resource is either an in-process timer (dies with the
process) or an append-only log file written one closed-handle line at a
time.

## Manual verification

Wake-from-sleep behavior is awkward to unit-test honestly without
stubbing out `NSWorkspace.shared.notificationCenter` and the
`NSApplicationDelegate` lifecycle — the production behavior is "AppKit
posts the notification, our block runs on the main queue, the polling
driver re-reads `state.json`." There is no domain logic that benefits
from mocks.

The manual test is therefore documented in the PR body for P2.10:

> Ran the app from Xcode (⌘R). Confirmed pet rendered. Slept the laptop
> (lid close), waited ~30 seconds, woke (lid open). Observed pet state
> refreshed within ~1 second of wake — confirmed by changing the
> sandboxed `state.json` to a different floor state during sleep and
> observing the post-wake render reflect that change without the usual
> up-to-one-second tick lag.
