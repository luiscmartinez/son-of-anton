# P2.05 — MenubarRenderer (TS-developer field notes)

What a TS reviewer needs to know to read `MenubarRenderer.swift` and
`MenubarRendererTests.swift` honestly. Concepts that show up in this PR
only.

## `NSStatusItem` lifecycle — why we hold it strongly

```swift
var statusItem: NSStatusItem?
// ...
let item = NSStatusBar.system.statusItem(withLength: .variableLength)
self.statusItem = item
```

- `NSStatusBar.system.statusItem(withLength:)` returns a freshly registered
  menu-bar slot. The system status bar **does not** retain it for you — the
  moment your last strong reference disappears, the icon is removed from
  the menu bar.
- That is why `MenubarApp` keeps `var statusItem: NSStatusItem?` as a
  stored property and assigns the freshly created item into it inside
  `applicationDidFinishLaunching(_:)`. The app delegate is itself retained
  by `NSApplication.shared`, so the chain `app → delegate → statusItem`
  keeps the slot alive for the process lifetime.
- TS analogue: think of `NSStatusItem` like an opaque DOM node you
  appended into a host you don't own. If you drop your only reference to
  it, the host garbage-collects it out of the view.
- The renderer (`var renderer: MenubarRenderer?`) is held the same way
  for the same reason — its timer's closure is `[weak self]`, so without
  a strong reference from the delegate the renderer would deallocate on
  the next event loop turn.

## `@MainActor` and AppKit threading

```swift
@MainActor
final class MenubarRenderer {
    func update(state: ActivityState, visualMode: VisualMode) { ... }
}
```

- AppKit (and most Cocoa frameworks) require all UI mutations on the
  main thread. `NSStatusItem.button.image = ...` is one of those calls —
  setting it from a background thread is undefined behavior, sometimes
  silently broken, sometimes a crash.
- Swift Concurrency's `@MainActor` annotation gives the compiler enough
  information to enforce that. A `@MainActor`-isolated type can only be
  used from the main actor; calling its methods from elsewhere requires
  an `await` hop, and the compiler refuses to forget.
- TS analogue: a `@MainActor` class is roughly "this object only exists
  on a specific event loop — touch it from a different loop and the
  compiler tells you no."
- The renderer is `@MainActor`. Its `Timer` callback hops back onto the
  main actor explicitly via `Task { @MainActor in self?.tick() }`, since
  `Timer.scheduledTimer(...)` does not natively understand actors.

## `Timer` on the main run loop vs. `DispatchSourceTimer`

```swift
timer = Timer.scheduledTimer(withTimeInterval: interval, repeats: true) {
    [weak self] _ in
    Task { @MainActor in self?.tick() }
}
```

- `Timer.scheduledTimer(withTimeInterval:repeats:block:)` schedules the
  closure on the **current run loop** in `.default` mode. The menu-bar
  app's main run loop runs in the main thread, so the closure fires on
  the main thread for free.
- The `[weak self]` capture is critical. `Timer` retains its block, and
  the block would otherwise retain `self`, creating a retain cycle
  through the timer — a known AppKit pitfall. With `[weak self]`, the
  block holds only a weak reference, so the renderer can deallocate when
  the rest of the app drops it; when that happens, the next timer tick
  finds `self == nil` and quietly no-ops until the timer is invalidated
  in `deinit`.
- `DispatchSourceTimer` is the lower-level GCD primitive that the
  Foundation `Timer` ultimately wraps for free-running cases. It is
  more flexible (cancellable, queue-targeted, no run loop required) but
  also more verbose. The ticket explicitly mandates `Timer` on the main
  run loop here, both for clarity and because we want the timer paused
  automatically when the run loop is asleep (sleep/wake hardening is
  P2.10's territory).
- TS analogue: `Timer` ≈ `setInterval` on the main thread;
  `DispatchSourceTimer` ≈ a manually-managed `setInterval` with
  cancellation and target-thread control.

## `Timer` re-creation on state transition

The renderer **re-creates** the timer on state transition rather than
keeping one perpetual timer and indexing into the active row.

```swift
private func restartTimer() {
    timer?.invalidate()
    let interval = 1.0 / Double(max(currentFrames.count, 1))
    timer = Timer.scheduledTimer(...)
}
```

- Frame interval is `1 / frameCount` seconds because the contract is
  1 second per full cycle. Different rows have different frame counts
  (`.idle` is 8, `.implementing` is 6, etc.), so the interval changes
  per state. The cleanest way to express "this row's interval" is to
  recompute it on transition.
- Re-creation also fits the "new state's loop begins from frame 0 on
  the next animation tick" contract from the ticket. Resetting
  `frameIndex = 0` and rescheduling the timer together makes the
  transition visually clean.
- Either pattern (re-create on transition or persistent timer + per-row
  indexing) is acceptable per the ticket; the renderer picks the
  re-create pattern and commits to it.

## `CIFilter`-based image desaturation

```swift
let filter = CIFilter.colorControls()
filter.inputImage = CIImage(cgImage: cg)
filter.saturation = 0
let output = filter.outputImage!
let outCG = ciContext.createCGImage(output, from: output.extent)!
return NSImage(cgImage: outCG, size: image.size)
```

- `CIFilter` is the Core Image image-processing pipeline. Filters are
  declarative nodes — you set inputs, read `outputImage`, and a
  `CIContext` renders the graph when you ask for concrete pixels.
- `CIColorControls` is the built-in saturation/contrast/brightness
  filter; setting `saturation = 0` produces a luminance-preserving
  grayscale (R, G, B all equal per pixel, weighted by Rec. 601 luma).
  The renderer uses that to paint the early failure visual described in
  the contract: same Mali frame, no color.
- `CIContext` is intentionally created once at renderer init and reused.
  Creating a new context per frame is expensive enough to be noticeable
  in the menu bar.
- The renderer applies the filter on-demand each frame. The other
  acceptable approach (per ticket) is a pre-computed desaturated frame
  cache at init. The on-demand approach was chosen because (a) only one
  state at a time animates, so the per-frame cost stays small, and
  (b) the code is simpler. If profiling shows it ever matters, the
  ticket Rationale section is where the swap to a cache should be
  documented.
- TS analogue: `CIFilter` ≈ a Canvas2D filter chain (`ctx.filter = "saturate(0)"`)
  paired with a draw call to commit pixels; `CIContext` ≈ the GPU/CPU
  backing the filter graph.

## Test seam — injected `ImageSink`

```swift
let renderer = MenubarRenderer(pet: pet) { image in
    lastImage = image
}
```

- The renderer accepts an injected closure (`ImageSink = (NSImage) -> Void`)
  rather than a real `NSStatusItem` reference. Production code wires the
  closure to `statusItem.button?.image = $0`; tests wire it to capture
  the emitted frame.
- This lets `MenubarRendererTests` run under a plain `XCTestCase`
  without a full `NSApplication` event loop, and without needing a
  visible menu bar item in CI.
- The renderer also exposes a small surface of `*ForTesting` properties
  (`currentStateForTesting`, `currentFrameIndexForTesting`, etc.) and a
  deterministic `advanceFrameForTesting()` method so tests can assert
  state-transition semantics without polling the real `Timer`. These
  are intentionally named to read as test-only seams at the call site.
