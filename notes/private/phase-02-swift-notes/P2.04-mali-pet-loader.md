# P2.04 — MaliPet loader (TS-developer field notes)

What a TS reviewer needs to know to read `MaliPet.swift` and
`MaliPetTests.swift` honestly. Concepts that appear in this PR only.

## `NSImage` vs. `CGImage` — two layers, same pixels

```swift
let image: NSImage = NSImage(contentsOfFile: path)!
let cg: CGImage = image.cgImage(forProposedRect: nil, context: nil, hints: nil)!
```

- `NSImage` is the AppKit-level abstraction: it can hold **multiple
  representations** (different sizes / DPIs) of the same image, draw into
  arbitrary `NSGraphicsContext`s, and lazily resolve which representation
  to use. Think of it as a high-level container.
- `CGImage` is the lower-level Core Graphics bitmap: a fixed-size pixel
  buffer with a colorspace and bit depth. No drawing, no autoscaling. It
  is the layer that lets you actually crop, copy, or read pixel sizes.
- `image.cgImage(forProposedRect:context:hints:)` asks the `NSImage` to
  produce a backing `CGImage` for a given target rect. `nil` for all three
  arguments means "any rect, no context, no hints" — fine for a static
  asset that has exactly one representation.
- TS analogue: `NSImage` ≈ a high-level `HTMLImageElement` (multi-DPI
  capable, draw-into-canvas friendly); `CGImage` ≈ the decoded
  `ImageBitmap` you crop pixels from.

The loader stores both: `spritesheet: NSImage` for downstream rendering,
`cgSheet: CGImage` for slicing.

## `CGImage.cropping(to:)` — `CGRect` is top-left origin

```swift
let rect = CGRect(x: col * w, y: row * h, width: w, height: h)
let slice: CGImage? = cgSheet.cropping(to: rect)
```

- `CGImage.cropping(to:)` returns a **view** over the source pixels, not a
  copy. The new `CGImage` shares the underlying buffer with the parent,
  so slicing 32 frames does not multiply memory by 32.
- `CGRect` coordinates here are **top-left origin** in image-pixel space.
  Row 0 at `y = 0` is the **top** of the spritesheet. This matches the
  grid PNG numbering, where row 0 is the top row.
- The **Cocoa flipped-coordinate gotcha** lives elsewhere: it bites when
  you draw into an `NSGraphicsContext` (whose Y axis runs bottom-to-top
  by default). It does **not** apply to `CGImage.cropping(to:)`. If
  P2.05's renderer flips frames vertically, suspect the draw path, not
  the slice.
- Returns `Optional<CGImage>` because cropping outside the source bounds
  yields `nil`. The loader's grid-compatibility check upfront keeps the
  per-frame rect strictly inside the sheet, so a `nil` here would be a
  real bug, not a normal case.

## Slicing approach in TS terms

```swift
for col in 0..<spec.frameCount {
    let rect = CGRect(x: col * frameWidth,
                      y: spec.rowIndex * frameHeight,
                      width: frameWidth, height: frameHeight)
    if let slice = cgSheet.cropping(to: rect) {
        out.append(NSImage(cgImage: slice, size: pixelSize))
    }
}
```

- Equivalent TS: walk `col` from `0..<frameCount`, pull a sub-rect out of
  a single decoded image, wrap each sub-rect in a renderer-friendly type.
- `0..<n` is a half-open Swift range — `[0, n)` in JS/TS terms. Use this
  form, not `0...n`, to avoid off-by-one.
- `NSImage(cgImage:size:)` wraps a `CGImage` back into an `NSImage` at a
  declared point size. `size` here is in **points**, not pixels — for
  this loader we pass pixel dimensions because the sheet is already at
  1×; on Retina the renderer is responsible for any additional scaling.
- `frames(for:)` is intentionally returning `[NSImage]` rather than
  `[CGImage]` because P2.05's renderer talks to `NSStatusItem.button`,
  which expects `NSImage`.

## Row-map table — why hardcoded

The `[ActivityState: RowSpec]` table lives in code, not in `pet.json` or
a sibling rows file, for four reasons documented in the ticket:

1. only one pet exists in Phase 02;
2. no consumer needs the format extension yet;
3. the mapping is visible in code review;
4. Phase 06's multi-pet catalog is the right place to formalize a
   pet-portable rows convention.

Wrong row indices will animate the wrong character pose in P2.05's
menubar renderer — visually obvious and easy to fix in source. Wrong
frame counts will animate a shorter or longer loop — also visually
obvious, also easy to fix in source.

## WebP via `NSImage(contentsOfFile:)`

macOS 13+ Image I/O decodes WebP transparently, so the loader does not
ship a third-party WebP decoder. If a future macOS target loses Image
I/O WebP support (or this stops working in CI), the fallback path is
either `CGImageSourceCreateWithURL` (the ImageIO C API directly) or
converting the fixture spritesheet to PNG. Either path goes into the
ticket Rationale section, not silently into the loader.

## `Error` enum with named cases — pattern

```swift
enum MaliPetLoadError: Error, Equatable {
    case petJsonNotFound
    case spritesheetUnreadable
    // ...
}
```

- Swift error types are sum types: each case is a distinct failure
  identity callers can pattern-match on. TS analogue:
  `type Err = { kind: 'petJsonNotFound' } | { kind: 'spritesheetUnreadable' } | ...`.
- The test `testLoaderThrowsWhenDirectoryMissing` matches on
  `MaliPetLoadError.petJsonNotFound` directly, not on a string. That
  keeps the renderer's failure-visual mapping (P2.07) cheap and typed.
