### Actionable findings

**Finding 1 — `open_pr` gate silently skipped when `subagentReview=disabled`**

- **File:** `tools/delivery/cli-runner.ts`, lines 1215–1222
- **What is wrong:** `openPrTarget` lookup uses only `status === 'subagent_review_complete'` as its auto-detect fallback. When `subagentReview=disabled`, tickets skip the subagent phase and remain at `verified` when `open-pr` is called. `openPrTarget` resolves to `undefined`; the `if (openPrTarget)` guard skips `emitOpenPrGate`. `openPullRequest` proceeds normally (its own lookup also checks `verified`), but `gate.json` receives no `open_pr` event.
- **Which clause:** Invariant 1 broken; spec-permits-real-bug (correctness gap demonstrable with `subagentReview=disabled` config).
- **Concrete fix:** Mirror `openPullRequest`'s internal resolution. Change:
  ```ts
  state.tickets.find((t) => t.status === 'subagent_review_complete')
  ```
  to:
  ```ts
  state.tickets.find((t) => t.status === 'subagent_review_complete') ??
  state.tickets.find((t) => t.status === 'verified')
  ```
  Or extract the shared lookup into a helper used by both the gate and `openPullRequest`.

### Advisory Observations

- **`eventsForOpenPrCommand` name creates future confusion.** The function is still imported at line 76 for notification warnings, but its name evokes the retired NDJSON pathway. A future reader may try to retire this import as part of P17.04 cleanup without realizing it still serves the notifier. A brief inline comment at the call site (or a rename) would prevent a silent regression.

- **`emitSoaEventForOpenPr` signature retains `_projectRoot`** (the old NDJSON write root), now unused. The stub comment says it will be removed in P17.04. This is fine, but the unused parameter is a minor API smell worth noting for the P17.04 cleanup pass.

- **Partial-state atomicity is now the sole integrity model for the four gates.** There is no rollback or correlation between gate.json and the primary action outcome. For `open_pr` specifically, a GitHub API failure after gate emit results in `gate.json` showing `open_pr` claimed for a ticket that has no PR. Downstream consumers of `gate.json` (e.g., codogotchi renderer) should treat `open_pr` as "attempted," not "succeeded." If any consumer currently treats it as confirmation of PR existence, that assumption is now unsound. This is an architectural property, not a P17.03 bug, but worth documenting in the gate spec.
