### Actionable Findings

**Finding 1 — `open_pr` gate skipped when ticket is at `verified` or `in_review`**
- **File:** `tools/delivery/cli-runner.ts`, lines 1215–1223
- **What is wrong:** `openPrTarget` uses only `status === 'subagent_review_complete'` as the no-arg fallback. `openPullRequest` (line 2071–2073) accepts `subagent_review_complete`, `verified`, or `in_review`. When the ticket is at `verified` (post `--ack-reconciliation`) or `in_review` (subagent disabled), `openPrTarget` is `undefined`, the gate is silently skipped, and `openPullRequest` still executes — **breaking Invariant 1.**
- **Clause:** Invariant 1 / spec clause "emit-then-action."
- **Concrete fix:** Mirror `openPullRequest`'s lookup order in the `openPrTarget` resolution:
  ```ts
  const openPrTarget =
    (parsed.positionals[0]
      ? state.tickets.find((t) => t.id === parsed.positionals[0])
      : (state.tickets.find((t) => t.status === 'subagent_review_complete') ??
         state.tickets.find((t) => t.status === 'verified') ??
         state.tickets.find((t) => t.status === 'in_review'))) ?? undefined;
  ```
  This ensures `openPrTarget` resolves to the same ticket that `openPullRequest` will act on, so the gate always fires before the action in all eligible status paths.

### Advisory Observations

- **`eventsForOpenPrCommand` survivor risk:** This function is still imported and called post-P17.03 as the input to `emitNotificationWarnings`. If a future ticket modifies this function and accidentally re-introduces NDJSON emission through the notification path, the test in p17-03 would not catch it (the test only exercises `emitSoaEventForOpenPr` directly). A narrow integration test that calls the full `open-pr` case and asserts no NDJSON side-effect would give stronger regression coverage, but this is style/test-depth, not a correctness gap today.

- **`emitSoaEventForOpenPr` retained for "backwards-compatible call sites" (comment, line 2358):** The comment implies external call sites exist, but a grep of the test suite shows only p17-03 importing it — and only to assert no-op behavior. If no active caller depends on the function body, the "backwards-compatible" framing is misleading. This is benign for P17.03 but could cause confusion during the P17.04 removal pass.

- **Gate conditionality pattern inconsistency:** Three of the four handlers (`adversarial_review`, `poll_review`, `record_review`) look up `target` before the `if (target) { emit; }` guard and throw or skip when not found. `open_pr` uses an `if (openPrTarget)` guard but never throws when `openPrTarget` is `undefined` — the action proceeds regardless. This asymmetry is what enables the Surface 2 bug. If the convention were "throw when target not found," the gate omission would be obvious immediately.
