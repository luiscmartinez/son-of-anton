### Actionable findings

**`tools/delivery/test/p17-04.test.ts`** — The doc-only `poll-review` fast path (outcome `'skipped'`) is not covered. After `recordReview(..., 'skipped', ...)`, the state has `reviewOutcome: 'skipped'`; `eventsForPollReviewCommand` on that state would produce a `review_recorded` event with `outcome: 'skipped'`. `emitReviewCleanGate` correctly does not fire (filter fails), but this case is untested. **Clause: (3) spec-permits-real-bug** — if `outcome === 'skipped'` were ever accidentally mapped to `'clean'` upstream, there would be no test catching a spurious gate write on the skipped path.

**Fix:** Add one test case in the poll-review describe block:
```ts
it('does not write gate.json when poll-review outcome is skipped (doc-only fast path)', async () => {
  const ticket = makeTicket(TICKET_ID, { status: 'reviewed', reviewOutcome: 'skipped' });
  const state = makeState(PLAN_KEY, [ticket]);
  const events = eventsForPollReviewCommand(state, TICKET_ID);
  await emitReviewCleanGate(events, enabledConfig(), PLAN_KEY);
  expect(existsSync(join(home, 'gate.json'))).toBe(false);
});
```

### Advisory Observations

- The `needs_patch` non-clean case is tested only for the record-review path. The poll-review and triage paths lack equivalent negative tests (outcomes `needs_patch`, `patched`, `operator_input_needed`). Parity would strengthen the test contract but is not required — the single function under test makes path-specific negative tests redundant given the implementation is a single filter on the event list.
- `eventsForReconcileLateReviewCommand` at notifications.ts:270 guards on `ticket.status !== 'done'` before building the event. A triage caller that passes a non-`done` ticket gets an empty events array and no gate write — correct behavior, but no test covers this edge.
- The doc-only poll-review branch (`docOnlyState`) uses `state.planKey` (pre-transition) rather than `docOnlyState.planKey`. These are identical in practice (planKey is immutable on a state object), but the asymmetry with `nextState` usage on the regular poll path (line 1297 uses `nextState` for events, line 1299 uses `state.planKey` for the gate) is a minor inconsistency that could confuse a future reader.
