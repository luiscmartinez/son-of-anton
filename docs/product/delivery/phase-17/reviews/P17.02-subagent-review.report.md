### Actionable Findings

**Finding 1 ‚Äî Invariant 2 broken when `previousState.tickets` has pending ticket before in_progress ticket**

- **File:** `tools/delivery/cli-runner.ts`, lines 2356‚Äì2382 (`emitGateForTransitions`)
- **File:** `tools/delivery/test/p17-02.test.ts`, line 150 (cook-mode test)
- **What is wrong:** `emitGateForTransitions` iterates tickets in array order without sorting or separating the two gate types. For a two-transition advance in cook mode, if the pending ticket (becoming in_progress) is at a lower index than the in_progress ticket (becoming done), the emit sequence is `ticket_started` then `ticket_completed`, leaving `ticket_completed` as the resident gate ‚Äî the inverse of the required invariant. The cook-mode test only exercises the `[in_progress_ticket, pending_ticket]` ordering, which happens to be correct, and does not catch the reversed case.
- **Invariant:** Invariant 2 (`ticket_started` must be the resident gate in cook mode).
- **Concrete fix:** In `emitGateForTransitions`, emit all `ticket_completed` events in a first pass and all `ticket_started` events in a second pass, regardless of array order. This makes the ordering invariant structural, not incidental:

```typescript
// First pass: emit completions
for (const previousTicket of previousState.tickets) {
  const nextTicket = nextState.tickets.find(t => t.id === previousTicket.id);
  if (previousTicket.status !== 'done' && nextTicket?.status === 'done') {
    await writeGateEvent(config, { gate: GATE_NAMES.TICKET_COMPLETED, ... });
  }
}
// Second pass: emit starts (last-write-wins = ticket_started)
for (const previousTicket of previousState.tickets) {
  const nextTicket = nextState.tickets.find(t => t.id === previousTicket.id);
  if (previousTicket.status !== 'in_progress' && nextTicket?.status === 'in_progress') {
    await writeGateEvent(config, { gate: GATE_NAMES.TICKET_STARTED, ... });
  }
}
```
Add a second cook-mode test with the pending ticket at index 0 to guard the regression.

**Finding 2 ‚Äî `redPolicy: undefined` silently emits `green_tdd` instead of `red_tdd`**

- **File:** `tools/delivery/cli-runner.ts`, lines 2385‚Äì2398 (`emitStartExitGate`)
- **What is wrong:** `ticket.redPolicy === 'required'` evaluates to `false` for any non-`'required'` value, including `undefined`. A state file written before `redPolicy` was introduced (or a partial state migration) would silently produce `green_tdd` for a ticket that should be guarded with `red_tdd`. No error is thrown; the codogotchi display would show TDD as satisfied when it is not.
- **Finding discipline:** Correctness gap demonstrable when state migration is incomplete (the TypeScript type excludes `undefined`, but runtime state files are not re-validated on read).
- **Concrete fix:** Tighten the conditional to a positive check for both valid values or add a guard:

```typescript
const gate =
  ticket.redPolicy === 'required'
    ? GATE_NAMES.RED_TDD
    : ticket.redPolicy === 'skip'
      ? GATE_NAMES.GREEN_TDD
      : GATE_NAMES.RED_TDD; // conservative default: unknown policy ‚Üí require TDD
```

This makes an unparsed `redPolicy` default to the stricter gate rather than silently skipping it.
