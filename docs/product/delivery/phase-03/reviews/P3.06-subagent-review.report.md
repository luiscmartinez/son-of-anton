### Findings

#### F1 — `CODOGOTCHI_DEMO_FRAME_MS=0` boundary not tested `[minor / no patch required]`

The spec says "positive integer" and `value > 0` correctly rejects `0`. However no test covers `CODOGOTCHI_DEMO_FRAME_MS=0 → 500`. The four existing env-var tests cover negative, non-numeric, absent, and valid-positive. The zero boundary is the only uncovered spec-reachable path. Not a correctness gap since the guard is right, but a test gap against the spec's "positive integer" language.

**Recommendation:** add `testDemoFrameMsZeroValueFallsBackTo500`.

---

#### F2 — Ticket spec cycle-order comment uses mixed separator style `[doc drift / informational]`

The ticket Review Focus section lists the cycle as:
> `idle`, `implementing`, `running-tests`, `reviewing`, `pushing`, `hyped`, `focused`, `nervous`, `waiting`, `celebrating`, `ascended`, `calling_for_backup`, `panicking`, `requesting_input`, `errored`

Notice `calling_for_backup` and `requesting_input` use underscores in that list but `running-tests` uses a hyphen — matching each state's raw value in the enum. The cycle array in code uses the correct Swift enum cases (`.callingForBackup`, `.requestingInput`), so there is no code bug. This is doc inconsistency only.

---

#### F3 — `demoFrameMs` called with a second `ProcessInfo` lookup instead of re-using the launch env `[style / no patch required]`

`MenubarApp.applicationDidFinishLaunching` already calls `DemoConfig.forLaunch()` which reads `ProcessInfo.processInfo.environment`. The subsequent `DemoConfig.demoFrameMs(from: ProcessInfo.processInfo.environment)` (line 113) re-reads the same immutable dictionary. No behavioral consequence — `ProcessInfo.environment` is stable — but the pattern is slightly inconsistent with the injected-env approach used everywhere else. A future change that wanted to make `demoFrameMs` part of the `DemoConfig` struct would clean this up naturally.

---

#### F4 — `testCycleDriverCycleContainsAllActivityStates` covers the same territory as `testCycleDriverExposes15StatesInRotation` `[redundancy / informational]`

`testCycleDriverExposes15StatesInRotation` checks `count == 15`. `testCycleDriverCycleContainsAllActivityStates` checks `allCases` membership. The second test is strictly stronger (a 15-element cycle with a duplicate would pass the count test but fail the membership test). The count test is therefore redundant. Not harmful, but worth noting if test-suite minimalism is a priority.
