## Findings

### F-1 · Missing log for spritesheet-specifically-absent soft degrade [correctness gap — untested]

**Ticket spec:** "Missing codogotchi sheet … the renderer logs the absence once (no chatty repeat)."

**What the code does:** `MenubarApp.swift:100–104` logs `"CodogotchiPet not available"` only when `try? CodogotchiPet()` returns `nil` — i.e., when `CodogotchiPet.init` throws. However, a missing spritesheet does **not** throw; `init` succeeds with `spritesheet = nil` (soft degrade). In that scenario — pet directory present, `pet.json` parseable, `codogotchi-spritesheet.webp` absent — the initializer returns a live `CodogotchiPet?` object, `MenubarApp` sees non-nil, skips the log, and the renderer silently returns empty frames for all nine codogotchi states with no log anywhere.

Neither `CodogotchiPet.init` (lines 91–97) nor `MenubarRenderer.resolveFrames(for:)` (lines 132–148) emits a log for the empty-frames case. The test `testMissingSheetSoftDegrades` verifies only non-crash and empty frames — no log assertion.

**Impact:** The operator running the menubar app with a pet directory populated by `pet.json` but missing the spritesheet will see silently idle animations for all nine states with no diagnostic. The ticket's "once" log requirement is unmet.

**Suggested patch** (smallest fix — at init time, one `NSLog` on soft degrade):

```swift
// CodogotchiPet.swift, after line 97 (the `return` in the soft-degrade branch)
NSLog("CodogotchiPet: spritesheet absent at %@ — codogotchi-owned states will render as idle", sheetURL.path)
return
```

The `testMissingSheetSoftDegrades` test should gain a `XCTAssertTrue(pet.spritesheet == nil)` assertion to document the soft-degrade signal. A log assertion in unit tests is impractical without a logger injection seam; a comment or separate integration note is sufficient for now.

### F-2 · Frame source-rect test verifies aspect ratio, not cell coordinates [test gap vs. Red spec]

**Ticket Red spec:** "Write a test that … each frame's source rect matches the expected cell coordinates."

**What the test does:** `testFramesForPanickingHaveCorrectSourceRect` (lines 92–110) checks:
- `first.image.size.height == 22` ✓
- Frame aspect ratio matches `sourceCellWidth / sourceCellHeight` ✓

It does **not** assert that panicking frames come from **row 6** rather than row 0 or any other row. A bug that always slices from `y = 0` (row 0) would produce frames with the correct aspect ratio and display height — the test would pass while returning celebrating frames for panicking.

**Impact:** The frame-extraction contract from the Red spec is partially enforced. The cell-coordinate assertion is missing.

**Suggested patch:** Compare pixel content between `frames(for: .panicking)` and `frames(for: .celebrating)` (different rows) to assert they are not identical. Alternatively, expose a test seam on `CodogotchiPet` that returns the `CGRect` used per frame and assert `rect.origin.y == 6 * (sheetHeight / 9)` for panicking. The simplest approach — extract first pixel (R,G,B) from row-0 frames and row-6 frames and assert they differ — is sufficient without adding a new seam.

### F-3 · Renderer Codex timer formula duplicates `MaliPet.frameInterval` [doc/code drift — not a correctness bug today]

**Ticket Review Focus:** "Adding a third sheet in a future phase should require touching one place, not multiple."

`MaliPet.frameInterval = 1.0 / Double(gridColumns) = 0.125 s` (MaliPet.swift:87).
Renderer line 197: `interval = 1.0 / Double(max(currentFrames.count, 1))`.

These are numerically equal for all current Codex rows (8 frames). However, the renderer reimplements the formula rather than reading `MaliPet.frameInterval`. If a future phase adds a Codex row with a different frame count, the renderer's formula yields a different interval than `MaliPet.frameInterval`. The ticket places timing in the loader, but the Codex timing does not flow from the loader to the renderer through the enum — only `CodogotchiPet.frameInterval` is consulted via `SpriteSource.codogotchi`.

**No patch needed in Green** (per ticket spec "smallest change that makes failing tests pass"). Worth flagging for P3.04 Refactor or P3.05: replace the inline formula with `MaliPet.frameInterval` for `.codex` and `.idleFallback` cases.

### F-4 · No overlap-guard between `MaliPet.rowMap` and `CodogotchiPet.rowMap` [untested invariant]

**Ticket Review Focus:** "Any state mapped in both sheets (none exist today by contract, but verify) renders from Codex per resolution order."

There is no test asserting `Set(MaliPet.rowMap.keys).isDisjoint(with: Set(CodogotchiPet.rowMap.keys))`. This is currently true by inspection — `.celebrating`, `.hyped`, etc. do not appear in `MaliPet.rowMap` — but the absence of a guard means a future `MaliPet.rowMap` addition of, say, `.reviewing` would silently shadow the codogotchi sheet entry without any test breakage.

**Suggested patch:** Add one test:

```swift
func testMaliPetAndCodogotchiRowMapsAreDisjoint() {
    let overlap = Set(MaliPet.rowMap.keys).intersection(Set(CodogotchiPet.rowMap.keys))
    XCTAssertTrue(overlap.isEmpty, "sheets must not share states: \(overlap)")
}
```

This is a gate-2 correctness invariant and should be in `MenubarRendererTests` or a dedicated cross-loader suite.
