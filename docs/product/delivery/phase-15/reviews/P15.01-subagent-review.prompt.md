### Attack surfaces to probe

1. **`appendSoaEvent` gate condition** — `config.codogotchi?.enabled === false` check: does it correctly handle `config.codogotchi` being `undefined` (should write), `{ enabled: true }` (should write), and `{ enabled: false }` (should skip)? 2. **`parseCodogotchiConfig` validation** — does passing an object with `enabled: undefined` (explicitly set to undefined) result in `enabled: true` via `obj['enabled'] !== false`? What about `enabled: null`, `enabled: 0`, or `enabled: "true"`? Are invalid values properly rejected?
3. **`resolveOrchestratorConfig` default** — `raw.codogotchi ?? { enabled: true }`: when raw has `codogotchi: { enabled: true }`, is it passed through as-is from `loadOrchestratorConfig`? Does the null-coalescing miss any falsy value for `raw.codogotchi`?
4. **`appendSoaEvent` catch scope** — the outer try/catch wraps `mkdirSync` + `open` + `fh.write` + `fh.close`. If `fh.close()` throws after a successful `fh.write`, is the data still flushed? Is there a double-write risk on retry?
5. **NDJSON line format** — `JSON.stringify(event) + '\n'`: if the event object contains a circular reference or non-serializable value, `JSON.stringify` throws synchronously inside the try block. Is that caught correctly?
