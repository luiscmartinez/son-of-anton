### Diff context

The diff introduces `maybeEmitReviewCleanRecorded` in `soa-event-feed.ts`:
```ts
export async function maybeEmitReviewCleanRecorded(
  events: DeliveryNotificationEvent[],
  config: ResolvedOrchestratorConfig,
  projectRoot: string,
): Promise<void> {
  const reviewEvent = events.find(
    (e): e is Extract<DeliveryNotificationEvent, { kind: 'review_recorded' }> =>
      e.kind === 'review_recorded' && e.outcome === 'clean',
  );
  if (reviewEvent) {
    await appendSoaEvent(
      config,
      projectRoot,
      buildSoaEventLine('review_clean_recorded', {
        plan_key: reviewEvent.planKey,
        ticket_id: reviewEvent.ticketId,
      }),
    );
  }
}
```

In `cli-runner.ts`, the three command handlers each capture their notification events to a local variable and call `maybeEmitReviewCleanRecorded` immediately after `emitNotificationWarnings`:

- `record-review`: `const recordEvents = eventsForRecordReviewCommand(nextState, ticketId); await emitNotificationWarnings(notifier, cwd, recordEvents); await maybeEmitReviewCleanRecorded(recordEvents, context.config, cwd);`
- `poll-review` (both doc-only and normal branches): same pattern with `docOnlyEvents` / `pollEvents`
- `triage-ticket`: `const triageEvents = eventsForReconcileLateReviewCommand(nextState, ticketId); await emitNotificationWarnings(notifier, cwd, triageEvents); await maybeEmitReviewCleanRecorded(triageEvents, context.config, cwd);`

The `codogotchi.enabled` gate lives inside `appendSoaEvent` — `maybeEmitReviewCleanRecorded` does not short-circuit before calling it.
