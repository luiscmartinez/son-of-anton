**Actionable findings**

**Finding 1 — Silent data loss on unreadable `app-state.json` (discipline clause 3)**
- File: `apps/menubar/Sources/AppState.swift:54`
- What is wrong: `try? Data(contentsOf: appStateURL())` treats `EPERM`, `EACCES`, and all other I/O errors identically to a missing file. A user whose `app-state.json` exists but is unreadable (e.g., after a permission change or sandboxing edge case) silently receives default state and their next `save` call overwrites the file they cannot read. No diagnostic is emitted.
- Clause: Spec-permits-real-bug — ticket says "missing or malformed → fallback" without distinguishing I/O errors; the behavior is spec-compliant but discards valid persisted state silently.
- Fix recommendation: Log a warning (OSLog at `.fault` level) when `Data(contentsOf:)` throws an error that is not `NSFileNoSuchFileError` / `ENOENT`, before returning fallback. This preserves the no-crash contract while making silent data loss visible in Console.app.

**Finding 2 — Future schema version fallback is untested**
- File: `apps/menubar/Tests/MenubarTests/AppStateTests.swift`
- What is wrong: The `payload.schemaVersion <= APP_STATE_SCHEMA_VERSION` guard is the only mechanism that protects against a schema-version-2 file silently loading with version-1 semantics in a future downgrade scenario. No test writes a file with `"schema_version": 2` and asserts the result equals the default state.
- Clause: Correctness gap — the branch is reachable in production (user upgrades, then downgrades) and is exercised by zero tests.
- Fix recommendation: Add `testFutureSchemaVersionFallsBackToDefault()` in `AppStateTests.swift` that writes `{"schema_version": 2, "floating_pet": {...}}` via `writeAppState(_:in:)` and asserts `state.isFloatingPetVisible == true` and `state.frame == FloatingFramePolicy.defaultFrame(in: visibleFrame)`.

**Advisory Observations**

1. **`withTempHome` environment mutation is process-global.** `setenv`/`unsetenv` modify the process environment, which is not thread-safe. If Xcode ever enables parallel test execution for this target, two tests mutating `CODOGOTCHI_HOME` concurrently could see each other's values. Tests always supply absolute tmp paths so this is not exercised, but a developer who sets a relative path in their environment would get a working-directory-relative store location that changes across launch contexts. An `isAbsolutePath` guard or documentation note would prevent confusion.
