Append (P2.01 as shipped):

- **xcodegen vs Xcode GUI template.** The Green section suggested using Xcode's "macOS App" template; shipped using [`xcodegen`](https://github.com/yonaskolb/XcodeGen) instead. `apps/menubar/project.yml` is the source of truth; `Menubar.xcodeproj` is generated and committed alongside it. Trade-off: adds one dev dependency (`brew install xcodegen`) but makes the project shape declarative, diffable, and reproducible — no hand-edited `pbxproj`, no Xcode-GUI bootstrap required to regenerate. The committed `.xcodeproj` still opens cleanly in Xcode and `bun run mac:test` works from a fresh terminal, satisfying the locked grill-me decision (`committed .xcodeproj`, Xcode-native `Info.plist` and entitlements). README documents `cd apps/menubar && xcodegen generate` as the regeneration command.
- **Info.plist generation.** Generated by xcodegen from the `info.properties` block in `project.yml` rather than hand-authored. `GENERATE_INFOPLIST_FILE: NO` in build settings ensures Xcode does not overwrite it. Only the five required keys (`LSUIElement`, `CFBundleName`, `CFBundleIdentifier`, `CFBundleShortVersionString`, `CFBundleVersion`) plus xcodegen's standard defaults are present. Reviewers verify the file diff, not the YAML.
- **Test host pattern.** `MenubarTests` is a `bundle.unit-test` target depending on `Menubar`. xcodegen wires `TEST_HOST` / `BUNDLE_LOADER` automatically; `@testable import Menubar` reaches `MenubarApp`. `@main` on the class does not interfere with instantiating it from the test.
