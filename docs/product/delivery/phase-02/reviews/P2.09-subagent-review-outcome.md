# P2.09 Swift Notes — Menu Items

Companion to `apps/menubar/Sources/MenubarMenu.swift`. Captures the AppKit
behavior a TS-leaning contributor would otherwise have to re-derive from
scratch.

## `NSMenu` / `NSMenuItem` in code (no `.xib` / `.storyboard`)

AppKit menus can be built two ways: in Interface Builder (drag items, wire
outlets, ship a binary-ish `.xib`/`.storyboard` in the bundle) or in pure
Swift (`NSMenu()` then `addItem(_:)` per `NSMenuItem`). Phase 02's framing is
"smallest reviewable Swift surface," so the menu is built in code. The
trade-off: no visual editor, but every menu change is a readable diff.

`NSMenuItem` is initialized with `title`, `action` (an Objective-C selector),
and `keyEquivalent` (a single-character shortcut, empty string for none).
There is no SwiftUI-style `Button { ... }` closure init for `NSMenuItem` —
the action surface is the classic target/action pair from 1989. That is why
this file uses `#selector(openLogFolder(_:))` instead of a trailing closure.

## Target/action wiring and the weak-target pitfall

When the user clicks an `NSMenuItem`, AppKit messages
`item.target.perform(item.action)`. The catch: **`NSMenuItem.target` is a
weak reference**. If the only strong reference to the target lives on the
stack of `applicationDidFinishLaunching`, ARC will deallocate it the moment
the launch callback returns, and every click after that will silently do
nothing. The menu still draws, the items still highlight on hover, but the
action selector is never invoked.

The fix in `MenubarMenu`:

- `MenubarApp` holds `var menuBuilder: MenubarMenu?` as a stored property
- `applicationDidFinishLaunching` assigns `self.menuBuilder = menuBuilder`
  before returning

The same pattern already protects `statusItem`, `renderer`, `demoDriver`,
`livePollingDriver`, and `transitionLog`. Adding another stored property is
the idiomatic AppKit answer to "why doesn't my menu item work?"

## `#selector` and `@objc` for closures-with-state

The action handlers live on `MenubarMenu` itself as `@objc` instance methods
(`@objc func openLogFolder(_:)`, `@objc func quitMenubar(_:)`). The `@objc`
attribute is required so the Objective-C runtime — which `NSMenuItem` uses to
dispatch — can find the selector. `MenubarMenu` must subclass `NSObject` for
the same reason: `#selector(...)` resolution needs the Obj-C runtime, not
just the Swift one.

The closures (`workspace`, `terminate`) are captured by `MenubarMenu` itself
and read by the `@objc` methods. This is the canonical way to bridge "I want
modern Swift closure injection" with "AppKit needs an Obj-C target/action
pair."

## `NSWorkspace.open(_:)` basics

`NSWorkspace.shared` is the macOS app-launching service — it knows which app
to use for a given URL (`Finder` for directories, `Preview` for PDFs, the
browser for `https://`, etc.). The relevant overload is:

```swift
@discardableResult
func open(_ url: URL) -> Bool
```

It returns `false` silently when the URL is not openable (file doesn't
exist, no associated app). For our case, `~/.codogotchi/` may not exist at
first launch — before the transition log or live polling driver has written
anything. We pre-create the directory with
`FileManager.createDirectory(at:withIntermediateDirectories: true)` before
calling `open`, so the menu action is never a silent no-op.

`createDirectory(at:withIntermediateDirectories: true)` is idempotent — it
does not error when the folder already exists. So the call is safe on every
invocation, not just first launch.

## Test seam: protocol over `NSWorkspace`

To assert "Open log folder triggers `NSWorkspace.open(_:)` against the
expected URL" without actually launching Finder, the menu takes a
`MenuWorkspaceOpening` protocol:

```swift
protocol MenuWorkspaceOpening: AnyObject {
    @discardableResult
    func open(_ url: URL) -> Bool
}

extension NSWorkspace: MenuWorkspaceOpening {}
```

`NSWorkspace.open(_:) -> Bool` is the exact selector signature the protocol
declares, so the extension is empty — Swift sees `NSWorkspace.shared` as
already conforming. Tests pass a `WorkspaceOpenSpy` that records the URL and
returns `true`. The same shape (narrow protocol + production-type
extension) is the standard Swift answer to "I need to mock a system class
without subclassing it."

## Termination spy

`NSApplication.shared.terminate(nil)` does not return cleanly from an
XCTest process — it actually attempts to terminate the test runner. So the
test passes a `terminate: () -> Void` closure that increments a counter,
and `MenubarMenu` stores that closure and calls it from `quitMenubar(_:)`.
The production default is `{ NSApplication.shared.terminate(nil) }`, so
behavior is unchanged for real launches.

via the `LivePollingDriver.setTooltip` closure. P2.09 does not change that
wiring, only confirms it. Unit tests cannot exercise menu-bar hover (no
window-server interaction in `xcodebuild test`), so the PR body for P2.09
records the manual verification: run the app, hover the menu-bar icon, see
the tooltip text change as the underlying state changes.
