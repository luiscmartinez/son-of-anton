### Invariants to hold

1. When demo mode is active, neither the app nor `DemoCycleDriver` reads from or writes to `~/.codogotchi/state.json`. The sandboxed `pollingTarget` URL must not resolve to that path under any environment combination handled by `DemoConfig.from(environment:arguments:)`.
2. `DemoCycleDriver.tickForTesting()` (and the production `start()` tick path) must emit the four floor states in exact order `idle → implementing → running-tests → celebrating` and wrap around to `idle` indefinitely. The `apply(state:)` callback must fire with the same state that corresponds to the fixture written to `sandboxedPath` on that tick (no off-by-one between bytes written and state emitted).
3. The atomic write at `sandboxedPath` must leave the file containing either the previous tick's bytes or the new tick's bytes — never a partial/empty file — even when the parent directory does not exist before the first tick.
4. With `CODOGOTCHI_DEMO` absent or set to any value other than `"1"`, and `--demo` absent from arguments, `DemoConfig.isDemoMode` is `false` and `pollingTarget` resolves to `~/.codogotchi/state.json`. No alias values (`"true"`, `"yes"`, `"on"`, `"1 "`, uppercase) activate demo mode.

### Attack surfaces to probe

1. `DemoConfig.from(environment:arguments:)` — environment parsing: `CODOGOTCHI_DEMO=""`, `CODOGOTCHI_DEMO="0"`, `CODOGOTCHI_DEMO="1\n"`, `CODOGOTCHI_DEMO="true"`, `CODOGOTCHI_DEMO="1 "` (trailing space). Does any non-"1" value silently activate demo mode? Conversely, can `CODOGOTCHI_DEMO=1` ever fail to activate when set?
2. `DemoConfig.from(...)` — `pollingTarget` resolution when `TMPDIR` is absent. Code falls back to `NSTemporaryDirectory()`. Does that always return a directory under which `appendingPathComponent("codogotchi-demo/state.json")` cannot collide with `~/.codogotchi/state.json`? What if `TMPDIR` contains `..` or is set to a literal `~`?
3. `DemoConfig.from(...)` — `--demo` argument matching uses `arguments.contains("--demo")`. If `arguments` includes only `["--demo=true"]` or `["--no-demo"]`, what happens? Conversely, do we accidentally match `--demo` anywhere later in argv even after an `--` argument separator?
4. `DemoCycleDriver.advance()` — index/state correspondence. The code increments `index` *before* the apply call (`index = (index + 1) % count`, then `apply(entry.state)`). Trace: at tick 1, entry = cycle[0] = idle, then index becomes 1, then apply(idle). Verify the index update cannot desync the bytes-written/state-emitted pair when the fixture read or write throws (does the next tick repeat or skip the current state?).
5. `DemoCycleDriver.advance()` atomic-write semantics. `Data.write(to:options:.atomic)` is documented to write to a sibling temp and rename. If the parent directory was just created with `createDirectory(at:withIntermediateDirectories:true)` and the process crashes between createDirectory and write, is the next start a clean recovery? If the sandbox path already exists as a directory (left over from a prior different run), does `.atomic` overwrite or throw?
6. `MenubarApp.applicationDidFinishLaunching(_:)` — demo wiring guard. The code requires `self.renderer != nil` before starting the demo driver. If `MaliPet()` fails (no pet assets on a fresh dev machine), demo mode silently no-ops with only an `NSLog` line. The ticket says "demo mode is active" should always cycle fixtures. Is the silent no-op acceptable, and does any test cover the demo-without-pet path?
7. `MenubarApp.applicationDidFinishLaunching(_:)` — `bundledDemoFixturesDirectory()` returns nil if `Resources/state-json/` is missing. Same silent NSLog no-op pattern. Verify the xcodegen `type: folder` change actually copies the directory shape into the built `.app` (not flattened), and the test build/xcodebuild test path agrees with the live launch path.
8. `MenubarApp.applicationWillTerminate(_:)` — the timer cleanup path. `demoDriver?.stop()` cancels the timer; deinit also invalidates. But if the renderer is captured `[weak]` in the apply closure and the renderer is deallocated before driver stop, does any in-flight tick crash by calling `update` on a deallocated renderer? (It uses `[weak renderer]` so should be safe — verify.)

### Diff context

DemoConfig is a pure struct with a `from(environment:arguments:)` static + `forLaunch()` production seam. DemoCycleDriver owns the timer, file write, and state apply per tick; its `start()` runs an immediate tick then schedules a 3-second repeating Timer. MenubarApp.applicationDidFinishLaunching now: resolves DemoConfig.forLaunch(), if isDemoMode AND renderer loaded AND fixtures bundle directory found, instantiates DemoCycleDriver with `apply: { [weak renderer] state in renderer?.update(state: state, visualMode: .normal) }` and calls `start()`. applicationWillTerminate calls `demoDriver?.stop()`. project.yml adds `Fixtures/state-json` as a `type: folder` resource so the directory ships inside the .app bundle Resources.

Tests use `tickForTesting()` to drive the driver synchronously, assert state sequence over 4 and 6 ticks, assert fixture bytes were written to a unique temp sandbox path, assert parent directory creation, and assert the four DemoConfig activation branches (default, env=1, --demo arg, env=0).
