### Attack surfaces to probe

1. `MenubarApp.applicationDidFinishLaunching` wake-observer registration — does the
   block-based observer get registered against `NSWorkspace.shared.notificationCenter`
   (correct workspace center) and not `NotificationCenter.default` (would silently
   no-op)? If the wrong center is used, the wake notification is never delivered and
   the post-wake refresh invariant is silently broken.
2. `MenubarApp.workspaceWakeObserver` lifetime — is the returned `NSObjectProtocol`
   token stored as a strong property on the delegate? If only held on the launch
   callback stack, ARC will release the token after launch returns and the
   notification center will release the block; the observer becomes a no-op.
3. `[weak self]` capture inside the wake block — is `self` weakly captured so the
   workspace center does not strongly retain `MenubarApp`? If the closure captures
   `self` strongly, the delegate cannot be deallocated and `applicationWillTerminate`
   cleanup behavior changes during process exit (less load-bearing here, but the
   pattern is a common foot-gun).
4. `applicationWillTerminate` removal ordering — is `removeObserver(_:)` called with
   the same token returned by `addObserver(forName:...)` and against
   `NSWorkspace.shared.notificationCenter` (not `NotificationCenter.default`)?
   Calling `removeObserver` on the wrong center leaves the block live on the
   workspace center; passing the wrong token type silently does nothing.
5. `applicationWillTerminate` re-entrancy — if the OS posts terminate twice in
   pathological shutdown paths, does the cleanup tolerate a nil `workspaceWakeObserver`
   on the second call? (The `if let` guard + nil-out should make it idempotent;
   verify.)
6. `LivePollingDriver.pollNow()` vs. `tickForTesting()` semantic identity — both
   call `runTick()` with no other state mutation. Does either function reset
   `lastRendered`/`lastTooltip`/`lastAgentState` caches in a way that would cause
   a duplicate transition-log entry or duplicate `apply` call when the wake-tick
   value matches the pre-sleep tick value? `runTick` reads those caches; an
   accidental cache clear would defeat the per-tick de-duplication that
   `LivePollingTests` asserts.
7. `pollNow()` thread-safety — `LivePollingDriver` is `@MainActor`. The wake
   observer registers `queue: .main`, so the block runs on the main run loop, but
   the closure call into the `@MainActor`-isolated `pollNow()` happens from a
   `NotificationCenter` block context. Does Swift concurrency complain at compile
   time, or does it actually need `MainActor.assumeIsolated` / `Task { @MainActor in ... }`?
   If `queue: .main` is not equivalent to MainActor isolation for the compiler, the
   build breaks or the call site reorders to a different actor.
8. Demo-mode interaction — in demo mode `livePollingDriver` is nil, so
   `livePollingDriver?.pollNow()` is a no-op. Is that the intended behavior? (Yes:
   demo mode runs `DemoCycleDriver` on its own 3s timer and a wake re-poll of the
   sandboxed file would just trigger an extra demo step. Confirm the diff does not
   accidentally call `demoDriver?.tick()` or similar.)
