### Attack surfaces to probe

1. `MenubarApp.applicationDidFinishLaunching` strong-ref of `menuBuilder` — is
   `self.menuBuilder = menuBuilder` reachable in every code path, or can an early
   return / guard / `MaliPet` failure path leave `menuBuilder` only as a stack
   local? (Invariant 2.)
2. `MenubarMenu.build` target wiring — both `openItem.target = self` and
   `quitItem.target = self` must be set before the items are added to the menu;
   confirm there is no path where one item is added without a target. Also confirm
   the action selectors named in `#selector(...)` exactly match the `@objc`
   instance methods declared on `MenubarMenu` (selector strings versus method
   names). (Invariant 1.)
3. `MenubarMenu.openLogFolder(_:)` directory pre-create — `try?` swallows every
   `FileManager.createDirectory` error. If creation fails (permission denied,
   path collision with a file), the subsequent `workspace.open(logFolderURL)`
   still runs and may silently fail. Is this acceptable given the Rationale's
   stated guarantee that the action "is never a silent no-op"? (Invariant 3 +
   spec-permits-real-bug clause: does the ticket Outcome / Rationale claim a
   stronger guarantee than the code delivers?)
4. `MenuWorkspaceOpening` protocol conformance via `extension NSWorkspace:
   MenuWorkspaceOpening {}` — `NSWorkspace.open(_:)` has multiple overloads (one
   taking `URL`, others taking arrays of URLs with configurations). Confirm the
   exact selector signature `func open(_ url: URL) -> Bool` resolves to the
   intended overload at call site and that production callers invoke the same
   selector the spy implements.
5. `MenubarApp` removed `NSApplication.terminate(_:)` selector wiring — when the
   user presses ⌘Q while the menu-bar app is the active app, does AppKit still
   route the standard Quit chord to `terminate(_:)`, or does the new `MenubarMenu`
   wiring (with `keyEquivalent: "q"` on a custom target) intercept it correctly
   such that the placeholder Quit behavior is preserved end-to-end?
6. Strong reference cycle — `MenubarMenu` holds a captured `terminate` closure
   whose default is `{ NSApplication.shared.terminate(nil) }`. `MenubarApp` holds
   `menuBuilder`. The default `terminate` closure does not capture `MenubarApp`,
   so no cycle exists in production. Confirm there is no test path where a
   capturing closure (e.g. `{ [weak self] in self?.terminate() }`) would create
   a leak if added later.

### Diff context

`MenubarApp.swift` removes the inline 6-line placeholder Quit construction and
replaces it with two lines: `let menuBuilder = MenubarMenu()` followed by
`item.menu = menuBuilder.build()` and `self.menuBuilder = menuBuilder`. A new
stored property `var menuBuilder: MenubarMenu?` joins the existing
strong-ref family (`statusItem`, `renderer`, `demoDriver`,
`livePollingDriver`, `transitionLog`). Nothing else in
`applicationDidFinishLaunching` is changed.

`MenubarMenu.swift` (new) declares the `MenuWorkspaceOpening` protocol with an
empty `NSWorkspace` extension, then a `final class MenubarMenu: NSObject` with
four init parameters (`workspace`, `terminate`, `logFolderURL`,
`fileManager`), a `static defaultLogFolderURL()` helper returning
`~/.codogotchi`, a `build() -> NSMenu` method that constructs two
`NSMenuItem`s with `target = self`, and two `@objc` action methods
(`openLogFolder(_:)`, `quitMenubar(_:)`). `openLogFolder` does
`try? fileManager.createDirectory(at:, withIntermediateDirectories: true)`
before `workspace.open(logFolderURL)`.

`MenuItemsTests.swift` (new) defines `WorkspaceOpenSpy` that records every
URL passed to `open(_:)`, plus three tests covering item count/titles,
`open(_:)` invocation with the configured URL, and termination-spy
invocation. Each test uses `target.perform(action, with: item)` to fire the
selector.
