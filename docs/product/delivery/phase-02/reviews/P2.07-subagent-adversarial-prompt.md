### Invariants to hold

1. Each `tickForTesting()` (and each production tick) emits at most one `apply(state, mode)` call. The emission occurs only when `(state, mode)` differs from the previously emitted pair; identical successive pairs are suppressed (`lastRendered` cache). On a transition from `(idle, normal)` to `(implementing, normal)`, exactly one new `apply` call is observed.
2. For every `StateReadError` case, the outcome is `(.idle, .desaturated)` with the exact canonical tooltip string from `LivePollingTooltips`. No failure case is ever silently rendered with `.normal` or with a `nil` tooltip; no success case is ever rendered `.desaturated`.
3. The `schemaNewer(got, expected)` tooltip interpolates both integers into the canonical template character-for-character. The literal trailing sentence `Update the menu bar app.` (period included) must remain present.
4. Live polling is mutually exclusive with demo mode at launch. `MenubarApp.applicationDidFinishLaunching` instantiates either `DemoCycleDriver` (demo branch) or `LivePollingDriver` (else branch), never both. `applicationWillTerminate` stops whichever driver was created.

### Attack surfaces to probe

1. `LivePollingDriver.emit(_:)` — `lastRendered` cache: when `apply` is called once with `(idle, desaturated)` (file-not-found), then the file appears with `activity_state: idle, schema_version: 1`, the next tick must emit `(idle, normal)` because `mode` changed even though `state` is unchanged. Verify the cache key compares both `state` AND `mode`, not just `state`.
2. `LivePollingDriver.emit(_:)` — tooltip cache: `hasEmittedTooltip` starts false. The first tick must always call `setTooltip` (even when tooltip is nil, to clear any inherited placeholder copy). Subsequent identical tooltips (including identical nils) must be suppressed. Verify a missing first emission cannot leak a stale tooltip from the placeholder image step earlier in `applicationDidFinishLaunching`.
3. `LivePollingDriver.decide(from:)` — switch coverage: every `StateReadError` case maps to `.idle + .desaturated`. If a future `StateReadError` case is added without updating `decide`, Swift's case-coverage check would catch it, but verify no `default:` arm exists that would silently swallow a new case.
4. `LivePollingDriver` initializer / start path — `start()` calls `stop()` then `runTick()` then schedules the timer. If `start()` is invoked twice on the same instance, can two timers race? Trace `stop()` semantics and confirm the timer reference is cleared before re-scheduling.
5. `LivePollingDriver.start()` — `Timer.scheduledTimer` is added to the **current** run loop. `MenubarApp.applicationDidFinishLaunching` runs on the main run loop, so this is fine for production. But the driver is `@MainActor`-isolated — confirm test invocation of `tickForTesting()` does not accidentally require the main run loop (tests are `@MainActor`, the call is direct, no timer involved — verify no hidden dependency).
6. `LivePollingDriver` retain-cycle handling — `Timer.scheduledTimer` retains its block, and the block captures `[weak self]`. The block hops to MainActor via `Task { @MainActor in self?.runTick() }`. Verify that `Task` capture does not silently strong-ref `self` through the closure environment.
7. `MenubarApp.applicationDidFinishLaunching` — `else if let renderer = self.renderer` arm. If `MaliPet()` fails (no pet assets), `self.renderer` is nil, the `else if` is skipped, and no live polling driver is created. The status item keeps the `pawprint` placeholder image; no tooltip is set. Verify this is the intended graceful degradation and not a regression vs. demo mode (which has the same behavior).
8. `MenubarApp` capture of `[weak item]` for tooltip closure — if `statusItem` is set to nil mid-lifecycle (unlikely but possible during shutdown), the closure no-ops. Verify nothing else holds the tooltip closure beyond the driver's `setTooltip` field.
