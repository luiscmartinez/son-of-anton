**2. JSONDecoder.convertFromSnakeCase on camelCase pet.json**
`[probed]` — The fixture `pet.json` uses camelCase keys (`displayName`, `spritesheetPath`). `convertFromSnakeCase` converts underscore-delimited keys to camelCase; it is a no-op on keys that are already camelCase (no underscores to process). No nil slices from off-by-one. Row 8 is safe.

**Actionable findings**

**Finding 1 — Silent frame-drop in `frames(for:)` not surfaced as error**
- File: `apps/menubar/Sources/MaliPet.swift:171`
- The `guard let slice = cgSheet.cropping(to: rect) else { continue }` silently drops nil slices. The Swift notes explicitly say "a nil here would be a real bug" — but the code treats it as a non-event. A future regression (frameCount exceeds real sheet width, broken CGImage backing) would return a shorter-than-expected array with no error signal, and all current tests would still pass (they only check `isEmpty`).
- Clause: Invariant 1 (frames for all states non-empty) could be broken silently; correctness gap — the code acknowledges it would be a bug but doesn't surface it.
- Recommendation: Replace `else { continue }` with `else { assertionFailure("CGImage.cropping returned nil for rect \(rect) — frameCount exceeds sheet bounds?"); continue }` so debug builds catch the regression immediately while release builds degrade gracefully with a non-empty partial result. Alternatively tighten the guard check to verify `col * frameWidth + frameWidth <= cg.width` before the loop.

**Finding 2 — `testLoaderThrowsWhenDirectoryMissing` accepts either error case**

**Findings for human review**

**A — Path traversal in `spritesheetPath`** (`MaliPet.swift:108`)
`manifest.spritesheetPath` is appended to `dirURL` without sanitizing `..` components or absolute-path prefixes. The fixture is clean. In production, if the pet directory is ever user-writable or served from an external source, a crafted `pet.json` with `spritesheetPath: "../../../../etc/sensitive.png"` would cause the loader to open a file outside the pet directory (limited to image-decodable files, but the behavior is still unexpected). Consider adding: `guard !manifest.spritesheetPath.hasPrefix("/"), !manifest.spritesheetPath.contains("..") else { throw MaliPetLoadError.petJsonMalformed }`. Low priority for Phase 02 (fixture is static), but worth fixing before user-supplied pet directories are supported.

**B — Grid-compatibility gate validates pixel size, not grid shape** (`MaliPet.swift:131-135`)
`cg.width >= 8, cg.height >= 9` only enforces minimum pixel dimensions. A 1200×900 spritesheet that is actually a 12×9 grid (not an 8×9 grid) passes the guard and silently produces misaligned slices. For Phase 02's single fixture this is fine; for Phase 06's multi-pet catalog it becomes a real risk. If the intent is to enforce the 8×9 grid more strictly, consider validating that `cg.width % 8 == 0 && cg.height % 9 == 0` and throwing `spritesheetIncompatibleGrid` on violation.

**C — `convertFromSnakeCase` is unnecessary on the camelCase fixture** (`MaliPet.swift:101-103`)
The strategy is a no-op on the current `pet.json` and adds silent permissiveness: a future snake_case `pet.json` would also decode without complaint. The strategy is harmless for Phase 02 but could mask schema drift in Phase 06. Human should decide whether to remove it (stricter, exact-match camelCase required) or document the permissiveness explicitly.

**D — Minor doc transposition in ticket Outcome** (`ticket-04-mali-pet-loader.md:18`)
The Outcome says "first element has the expected pixel dimensions (rowHeight × frameWidth)" — height before width. Should read "(frameWidth × frameHeight)" or "(width: frameWidth, height: frameHeight)". The test is correct; only the doc label is transposed.

**E — Production debuggability: all `Data` failures collapse to `petJsonNotFound`**
Permission-denied, I/O error, and file-not-found on `pet.json` all produce `petJsonNotFound`. In a production failure scenario where a pet directory exists but has wrong permissions, the rendered error visual ("pet.json not found") would be misleading. Not a Phase 02 concern (fixture tests cover the relevant paths), but worth noting before P2.07's renderer maps error cases to user-visible visuals.
