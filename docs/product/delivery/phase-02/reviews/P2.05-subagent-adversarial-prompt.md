### Ticket scope

P2.05 Swift MenubarRenderer — NSStatusItem + continuous-loop animation. Outcome:
`apps/menubar/Sources/MenubarRenderer.swift` exposes a `MenubarRenderer` type that
owns the `NSStatusItem` and is driven by `update(state: ActivityState, visualMode:
VisualMode)`. `VisualMode` has cases `.normal` and `.desaturated`. While a floor
state (`.idle`, `.implementing`, `.runningTests`, `.celebrating`) is held in
`.normal` mode, the renderer animates that state's frames (from
`MaliPet.frames(for:)`) on a 1-second-per-cycle continuous loop. On state
transition the new state's loop begins from frame 0 on the next tick. `.desaturated`
mode renders the current state (typically `.idle`) with reduced saturation via a
`CIFilter` (`CIColorControls` with `inputSaturation = 0`) — or, equivalently, a
pre-computed desaturated frame cache. All `NSStatusItem.button.image` writes happen
on the main thread (`@MainActor` or explicit dispatch). Frame interval = 1 second /
frameCount, scheduled via `Timer.scheduledTimer(withTimeInterval:repeats:)` on the
main run loop. The renderer does not read `state.json` (that is P2.07's job).
Tests at `apps/menubar/Tests/MenubarTests/MenubarRendererTests.swift` assert: after
`update(state: .implementing, visualMode: .normal)` the renderer's current frame
source is the implementing row's frames; after a second `update(state:
.runningTests, ...)` the renderer swaps to the running-tests row starting at frame
0; after `update(state: .idle, visualMode: .desaturated)` the rendered image is
visibly desaturated (pixel sample R == G == B or filter chain inspection). Swift
notes at `notes/private/phase-02-swift-notes/P2.05-menubar-renderer.md` explain
NSStatusItem lifecycle, `@MainActor` and AppKit threading, `Timer` vs.
`DispatchSourceTimer`, and `CIFilter`-based desaturation in TS-developer terms.

Rationale notes (delivery): timer pattern is re-create-on-transition (interval
recomputed each transition because frame counts differ per row); desaturation is
on-demand per-frame `CIFilter.colorControls()` with a single reused `CIContext`;
test seam is an injected `ImageSink = (NSImage) -> Void` closure plus a small
`*ForTesting` surface and a deterministic `advanceFrameForTesting()` method;
`MenubarApp` tries `MaliPet()` on launch and falls back to the placeholder
`pawprint` icon if pet assets are missing (logs the failure rather than crashing).

### Invariants to hold

1. **State transition resets frame index to 0.** After `update(state: A, visualMode: M)`
   followed by some number of timer ticks that advance `frameIndex`, a subsequent
   `update(state: B, visualMode: anyMode)` with `B != A` must set `frameIndex = 0` and
   swap the active frame source to `pet.frames(for: B)` before any further paint.
2. **Frame source matches active state.** While the renderer is holding state `S`,
   `currentFramesForTesting` must equal `pet.frames(for: S)` (same `count`, same
   per-frame pixel dimensions); the timer must not advance into another row.
3. **Desaturated mode produces grayscale output.** When the renderer is in
   `.desaturated` mode, every `NSImage` sent to the `ImageSink` must have R == G == B
   (within rounding tolerance) for every opaque pixel — never the original colored
   frame.
4. **Main-thread image writes.** Every write to the `ImageSink` (and therefore to
   `NSStatusItem.button.image` in production wiring) must occur on the main actor /
   main thread. No background-queue writes via the timer block.
5. **Timer ownership is sound.** The renderer must not leak through the
   `Timer.scheduledTimer` block: dropping the only strong reference to the renderer
   must allow it to deallocate and the timer must be invalidated (no stale timer
   firing into a freed renderer; no retain cycle through the block's `self`).

### Attack surfaces to probe

1. `MenubarRenderer.update(state:visualMode:)` — mode-only change path: when
   `state` is unchanged and only `visualMode` flips (`.normal` → `.desaturated`),
   does the renderer repaint the current frame under the new mode, or does it
   wait for the next timer tick? If it waits, the user sees a stale-color frame
   for up to `1 / frameCount` seconds — does that violate the ticket's
   intent for desaturation to take effect on update?
2. `MenubarRenderer.update(state:visualMode:)` — repeated same-state same-mode
   call: does it pointlessly invalidate and re-schedule the timer (skipping a
   tick and visually stalling the animation), or is it a no-op? The
   implementation uses `let stateChanged = state != currentState || currentFrames.isEmpty`
   — check whether `currentFrames.isEmpty` can ever be true after `init` so the
   guard does not silently always re-arm.
3. `MenubarRenderer.restartTimer()` — when the row frame count is 0 (which can
   happen if a future `ActivityState` is added without a row map entry), the
   computed `interval = 1.0 / max(frameCount, 1) = 1.0`. Combined with the
   `guard !currentFrames.isEmpty else { return }` in `tick()`, this yields a
   1Hz no-op timer — is that the intended degradation, or does it mask a real
   ticket-relevant bug (state-without-row silently animates nothing)?
4. `MenubarRenderer.desaturate(_:)` — the failure paths (`cgImage(forProposedRect:...)`
   returns nil, `filter.outputImage` returns nil, `createCGImage(_:from:)` returns
   nil) all return the **original colored image**. In `.desaturated` mode that is
   a silent invariant violation (Invariant 3): the user sees a colored frame and
   the renderer reports `.desaturated`. Is this an acceptable fallback for a
   ticket whose stated purpose is "early failure visual"?
5. `MenubarRenderer.restartTimer()` block — `Timer.scheduledTimer` retains its
   block; the block captures `[weak self]` and dispatches a `Task { @MainActor
   in self?.tick() }`. The `Task` itself captures `self` weakly through the
   `self?.tick()` indirection, but the `Task` closure still runs after the
   timer fires. Does the `[weak self]` actually let the renderer deallocate
   between the outer block firing and the `Task` running, given that the timer
   is the only thing keeping the renderer's `deinit` from invalidating itself?
   Probe the retain-cycle / use-after-free shape here directly against the
   posted spec ("renderer does not hold a strong reference cycle through the
   timer's closure" — ticket Review Focus).
6. `MenubarRenderer.advanceFrameForTesting()` — used by the red test to advance
   `frameIndex` deterministically before the state transition. Does it actually
   match the production tick path (`tick()`), so a passing test really exercises
   the same wraparound semantics that the real timer would? Probe whether the
   test seam diverges from production in any way that hides a bug.
7. `MenubarApp.applicationDidFinishLaunching` — the renderer's `ImageSink` is
   captured as `{ [weak item] image in item?.button?.image = image }`. Is
   `[weak item]` safe here? The `MenubarApp` already holds `statusItem`
   strongly, and `MenubarApp` itself is retained by `NSApplication.shared`. If
   the renderer outlives the status item somehow (e.g. status item is removed
   by future cleanup code), the sink silently no-ops — does that violate
   Invariant 4 (the test that "all writes happen on the main actor" is
   trivially satisfied if there are no writes at all)?
