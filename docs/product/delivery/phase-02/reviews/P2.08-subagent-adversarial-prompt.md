### Ticket scope

P2.08 ships `apps/menubar/Sources/TransitionLog.swift`, an append-only NDJSON writer
that:

- Appends one NDJSON line on every observed state change.
- Appends one heartbeat NDJSON line every 60 minutes if no state change has occurred
  in that window. Heartbeats reset on any real state change.
- Writes to `~/.codogotchi/state-transitions.log` by default; path overridable via DI
  for tests and demo mode (sandboxed location).
- Rotates the log when its size exceeds 10MB on a write: rename existing file to
  `state-transitions.log.1` (overwriting any prior `.1`), open a fresh log file.
  Single backup; no `.2`/`.3`.

NDJSON transition shape:
`{"ts":"...","state":"...","prev":"...","schema_version":1,"source_origin":"...","source_kind":"...","source_name":"..."}`

NDJSON heartbeat shape:
`{"ts":"...","state":"<last activity>","heartbeat":true,"schema_version":1}`

`source_origin`/`source_kind`/`source_name` are pulled from the snapshot's
`source_event` when available; null/absent when not.

The live polling driver (P2.07) and demo driver (P2.06) both call
`TransitionLog.recordTransition(snapshot:previousState:)` on observed state changes.
Heartbeats are timer-driven inside the log, not driven by the polling driver.

Rationale notes added by this delivery:

- **Heartbeat-vs-rotation accounting decision**: heartbeats *do* count toward the
  rotation byte threshold.
- **Heartbeat timer cadence**: a 60-second probe `Timer` consults the injected clock
  and only emits when the heartbeat interval has elapsed (resilient to sleep/wake).
- **`SourceEvent`** added to `StateSnapshot` + `StateJsonReader`. Field names: `origin`,
  `kind`, `name`. Optional everywhere.
- **Demo path**: demo mode writes its log under a sibling of the sandboxed
  `pollingTarget` so a live run is never trampled.
- **LivePollingDriver only records on `.success`** — failure visuals collapse to
  `.idle` regardless of agent state and would otherwise flood the log with phantom
  `prev=idle` entries during hook hiccups.

### Invariants to hold

1. `recordTransition(snapshot:previousState:)` writes exactly one well-formed
   NDJSON line per call (newline-terminated, parses with `JSONSerialization`),
   containing the keys `ts`, `state`, `prev`, `schema_version`, and the
   `source_origin`/`source_kind`/`source_name` triplet when `snapshot.sourceEvent`
   is non-nil.
2. After a `recordTransition`, no heartbeat line is emitted on a `tickHeartbeatForTesting`
   (or production timer probe) until `clock()` advances by at least `heartbeatInterval`
   beyond the last activity timestamp.
3. When the active log exceeds 10 MB and a transition triggers `write`, the file is
   renamed to `<path>.1` (overwriting any existing `.1`) and the new write lands in a
   fresh `<path>`; `<path>.2` is never created across repeated rotations.

### Attack surfaces to probe

1. `TransitionLog.write` — handle leak / partial-write on injected-clock variants:
   does the per-call `FileHandle(forWritingTo:)` always close, including on a thrown
   `seekToEnd()` or `write(contentsOf:)`?
2. `TransitionLog.rotate` — non-atomic window where `removeItem(.1)` succeeds but
   `moveItem(.log → .1)` fails: does the next write recreate `.log` cleanly, or does
   it leave the system in an unrecoverable state? Also: what if rotation runs
   concurrently with a heartbeat tick scheduled on the main run loop?
3. `TransitionLog.maybeEmitHeartbeat` reset semantics: after the first heartbeat
   fires, is `lastActivityAt` advanced so that the next heartbeat only fires another
   full `heartbeatInterval` later, not on every subsequent probe?
4. `LivePollingDriver.runTick` — when `lastAgentState` is nil (first tick) and the
   read succeeds, the code passes `previousState: snapshot.activityState` to
   `recordTransition`. Does this produce a self-referential `state == prev` log line,
   and is that the intended observability for "first observation"?
5. `DemoCycleDriver.advance` — the new code parses the just-written fixture via
   `StateJsonReader.read(at:)` to populate the snapshot for the log. If the parse
   fails (corrupt fixture, future schema-version drift), is the failure silent and
   does the demo cycle continue rendering states without any log entry?
6. `StateJsonReader` change — adding `sourceEvent` to `StatePayload` as
   `SourceEvent?`. If the payload's `source_event` field is present but malformed
   (e.g. an array, or `origin` non-string), does the whole read now return
   `.failure(.malformed)`, breaking previously-passing fixtures and live polling?
